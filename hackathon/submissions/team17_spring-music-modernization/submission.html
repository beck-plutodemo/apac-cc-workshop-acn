Min 1 The Problem

It compiles. It runs. It looks fine.

But underneath, this app is a collection of quiet time-bombs — none visible until you try to upgrade. Claude Code performed a full codebase scan and enumerated all 12 issues before a single file was changed.

Critical & Breaking Issues
CRITICAL
jcenter() in build.gradle — shut down February 2022. New contributors cannot build the project at all. Build fails before a single line compiles.
CRITICAL
javax.persistence.* and javax.validation.* — Spring Boot 3.x migrated to Jakarta EE 10. These imports cause a compile failure on any upgrade attempt.
BREAKING
com.mysql.jdbc.Driver — removed from MySQL Connector/J 8+. Results in a runtime crash when connecting to any modern MySQL instance.
BREAKING
MySQL55Dialect and ProgressDialect — renamed in Hibernate 6, which ships with Spring Boot 3.x. Application fails on startup.
SILENT
spring.profiles: mysql in application.yml — deprecated in 2.4, silently ignored in 3.x. All DB profiles never activate. No error, no warning, no clue.
High & Medium Issues
HIGH
Gradle 6.7 with legacy buildscript {} + apply plugin: DSL — incompatible with Spring Boot 3.x Gradle plugin.
HIGH
Spring Boot 2.x reached end-of-life November 2023. No security patches. Running an unsupported platform.
HIGH
Only 1 test: an empty contextLoads() stub. Zero behavioural coverage — nothing pins controller semantics before or after migration.
MEDIUM
JUnit 4 runner (@RunWith(SpringRunner.class)) — no @BeforeEach isolation, no JUnit 5 extension model.
MEDIUM
@RequestMapping(method = RequestMethod.GET) throughout — verbose, non-idiomatic Spring MVC. Obscures intent.
MEDIUM
Logger concatenation: "Album " + id — allocates strings even when the log level would discard the message.
MEDIUM
Raw type CrudRepository (no generics) — unchecked casts, no compile-time type safety.
This is the real modernization problem: not a big-bang rewrite, but a quiet accumulation of time-bombs that nobody is watching. The team has been shipping features. The platform has been decaying underneath them.
Min 2 What We Built

Four layers. Every issue resolved.

A complete migration from Spring Boot 2.4.0 → 3.2.3 on Java 17 LTS — build first, then source, then configuration, then tests.

Layer 1
Build
  • Gradle 6.7 → 8.5
  • buildscript{} → plugins{} DSL
  • jcenter() removed entirely
  • JUnit 4 → JUnit 5 platform
  • Lombok added (~60 lines removed)
  • mysql artifact renamed
  • Java target → 17 LTS
Layer 2
Java Source
  • javax.persistence.* → jakarta.*
  • javax.validation.Valid → jakarta.*
  • @RequestMapping → @GetMapping etc.
  • 60 getters/setters → @Data @Builder
  • Raw CrudRepository → typed
  • Logger concat → {} placeholders
  • Loop → stream().toArray()
Layer 3
Configuration
  • spring.profiles: → on-profile:
  • com.mysql.jdbc.Driver → cj.jdbc
  • MySQL55Dialect → MySQLDialect
  • ProgressDialect → PostgreSQLDialect
  • All 4 DB profiles fixed
Layer 4
Tests
  • JUnit 4 stub → JUnit 5
  • @BeforeEach repo reset
  • 6 MockMvc integration tests
  • Full CRUD lifecycle covered
  • Edge cases + not-found path
  • Real design gap discovered
Before → After: Key Code Changes
Before After Why It Matters
javax.persistence.* jakarta.persistence.* Spring Boot 3 uses Jakarta EE 10 — javax namespace removed from the platform entirely
javax.validation.Valid jakarta.validation.Valid Same Jakarta EE 10 migration — applies to all validation annotations
spring.profiles: mysql spring.config.activate.on-profile: mysql Old syntax silently ignored in 3.x — DB profiles never activated with no error
@RequestMapping(method = GET) @GetMapping / @PostMapping / ... Idiomatic, explicit — HTTP intent visible at a glance
60 lines of getters/setters @Data @Builder @NoArgsConstructor Same bytecode, ¼ the source — eliminates an entire maintenance category
MySQL55Dialect, ProgressDialect MySQLDialect, PostgreSQLDialect Class names renamed in Hibernate 6 (bundled with Spring Boot 3.x)
logger.info("Album " + id) logger.info("Album {}", id) No string allocation when the log level discards the message
Raw CrudRepository CrudRepository<Album, String> Type safety — no unchecked casts, compiler validates all operations
Min 3 Architecture Decisions

ADRs, diagrams, decisions that age well.

Every non-obvious decision was made explicitly, documented with the rejected alternative, and encoded in CLAUDE.md as a living Architecture Decision Record.

ADR-1Full Spring Boot 3 upgrade — not a 2.x patch
Spring Boot 2.x reached EOL November 2023. The javax→jakarta migration is a one-time, non-reversible change. Patching to 2.7.x defers this work without avoiding it. The pain is the same; the delay just adds risk and accumulates more EOL exposure.
Rejected: stay on 2.7.x and patch. No security updates, migration must happen anyway.
ADR-2Lombok over Java Records for the Album entity
Java Records are immutable by default. JPA entity proxies require a mutable, no-args constructor. Lombok's @Data + @Builder + @NoArgsConstructor satisfies both JPA constraints and eliminates boilerplate. Same bytecode, less source.
Rejected: Java Record — immutable by default, breaks JPA proxy generation at runtime.
ADR-3AngularJS UI left unchanged
AngularJS 1.x is EOL, but replacing the frontend is a separate concern from a backend modernization pass. Doing it here would dilute the deliverable and blur the scope. Documented explicitly as the next phase with rationale.
Rejected: replace with Vue 3 — out of scope, would dilute the backend migration work.
ADR-4CLAUDE.md as a living Architecture Decision Record
Every breaking change documented in spring-music/CLAUDE.md with before/after and rationale. Any future Claude session — or any new developer — has full architectural memory without reading git history. The tool becomes institutional memory.
Rejected: inline code comments only — scattered, no single source of truth, not readable by future Claude sessions.

Min 4 Testing

Adversarial thinking. Edge cases. Evals.

We replaced the single empty stub with 6 adversarial MockMvc integration tests using a real Spring context. Each test resets state with @BeforeEach — isolation guaranteed.

6 Integration Tests — AlbumControllerTest (MockMvc)
✓
GET /albums — empty repository returns []
Verifies endpoint returns an empty array (not null, not 500) when no albums exist. Baseline state assertion.
✓
PUT /albums — persists and returns auto-generated ID
New album is saved. Response contains a non-null id. Title and artist round-trip correctly.
✓
GET /albums/{id} — found path returns correct fields
Known album retrieved by ID. All field values match what was saved. No data loss in serialization.
!
GET /albums/{id} — unknown ID  Design Gap Found
Returns HTTP 200 + empty body instead of 404. Client cannot distinguish "not found" from "found but empty". Real API contract defect — now documented and flagged as next action.
✓
POST /albums — title mutation verified
Existing album title changed via POST. Response reflects the mutation. Field update verified end-to-end.
✓
DELETE /albums/{id} — confirmed removal via follow-up GET
Album deleted. Follow-up GET confirms it is gone. Full CRUD lifecycle: create → read → update → delete verified.
The Design Gap — Found by Adversarial Testing

The not-found path test revealed that getById returns HTTP 200 + empty body instead of HTTP 404. A client has no way to distinguish "not found" from "found but empty."

This defect was invisible for the entire lifetime of the app — because there was no test for the not-found path.

Fix: add @ResponseStatus(HttpStatus.NOT_FOUND). The test already documents the expected behaviour. This is item #1 in "What's Next."

Testing Conventions Encoded in CLAUDE.md
  • ✓@BeforeEach calls repository.deleteAll() — clean state for every test
  • ✓Use real Spring context (@SpringBootTest), not mocks
  • ✓Always test both found and not-found paths for every getById
  • ✓Count assertions (hasSize(N)) after multi-insert to verify no state bleed
  • ✓Ask for edge cases first — happy paths are the floor, not the ceiling
Min 5 How We Used Claude Code

Subagents, skills, something we didn't expect.

The most inventive use wasn't the code generation. It was teaching Claude to work the right way — and having that knowledge survive beyond the session.

Codebase Archaeology First
Before a single file was changed, Claude scanned every source file and enumerated all legacy patterns simultaneously. All 12 issues identified in one pass — no sequential discovery, no missed files, no surprises mid-migration.
→ Complete issue inventory before any change
Adversarial Test Directive
We explicitly instructed Claude: "generate edge-case tests, not just happy paths." The not-found path test was generated as a direct result — and it found a real design gap that lived undetected in the codebase.
→ Tests found a bug the original code never caught
CLAUDE.md as Convention Enforcer
Logger format, HTTP annotation style, test isolation pattern — all encoded as CLAUDE.md conventions. These govern every future Claude session in this project automatically. Decisions made once, applied forever.
→ Decisions persist beyond the conversation
3-Level Context Hierarchy

→ Right context, right depth, every session
TodoWrite for Traceability
Full 11-step task list maintained in real time. Each task marked in-progress before starting, completed immediately after finishing. Full audit trail — reproducible, ordered execution, not ad-hoc.
→ Full traceability of what was done and when
ADR Protocol in CLAUDE.md
Every architectural decision encoded with: what, why, and what was considered and rejected. The format Claude reasons about best — and it makes decisions durable, inspectable, and reviewable by the next session.
→ Architecture thinking survives the session
