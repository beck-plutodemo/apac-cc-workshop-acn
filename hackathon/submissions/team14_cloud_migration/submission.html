Challenge 01 — The Memo · PM

We pick a side: lift first, optimise fast.

The decision

Containerise all three workloads and deploy to ECS Fargate without refactoring. Get a cloud baseline running on real AWS infrastructure first. Then optimise on real production metrics — not on assumptions made in a meeting room.

Why not refactor on the way in?

Refactoring and migrating simultaneously doubles the blast radius of every mistake. We don't know which parts of the architecture are actually bottlenecks until we see production cloud metrics. Premature optimisation is just as dangerous in infrastructure as it is in code.

Risks we're accepting

  • → Phase 1 cloud costs will be higher than optimised state — accepted, known, time-bounded to 90 days
  • → Technical debt travels with us — mitigated by the parallel DR runbook and rollback plan at every stage
  • → Downstream consumers see no improvement on day one — mitigated by compatibility shim on recon output

The phased plan

Full test suite green locally and in AWS before any DNS cutover.

Challenge 02 — The Discovery

What was actually running

Browser └─→ Nginx TLS :443 [⚠ self-signed cert, key only on disk] └─→ webapp Node.js :3000 (Node 16 EOL) ├─→ postgres-app (10.0.1.44:5432) ├─→ Redis (10.0.1.46:6379) ⚠ NO AUTH └─→ NFS /mnt/reports (10.0.1.47) ⚠ not documented └─→ recon-job (cron 2am) └─→ reportingdb ⚠ HARDCODED 10.0.1.45 └─→ Metabase ⚠ NOT IN SCOPE — 5 team dashboards

Before we could move a single workload, we needed to understand what actually existed — not what the architecture diagram said. We found four undocumented dependencies: a Redis instance with no password, a TLS private key that existed only on a physical server, a hardcoded IP address in the reconciliation config, and a Metabase BI tool quietly querying the reporting database that nobody had included in the migration plan. Every one of these would have caused an outage if we'd proceeded without finding them.

The Architecture

Local Docker = Production AWS

Docker ServiceAWS EquivalentPurpose
postgres-appRDS PostgreSQL 15 (db.t4g.medium)App database
postgres-reportingRDS PostgreSQL 15 (db.r6g.xlarge)Reporting — 5 teams
redisElastiCache Redis 7 (cache.t4g.small)Session store
minioS3 (contoso-recon-input)Recon file drop
webappECS Fargate (2 tasks)Customer web app
recon-jobECS Fargate (event-triggered)Reconciliation processor
nginxApplication Load BalancerTLS + routing

The key principle here is that your local environment is your cloud. Every Docker service is named to its AWS equivalent, every environment variable maps directly to what ECS will inject in production, and every volume maps to EFS or S3. When you run docker compose up today and aws ecs deploy tomorrow, the behaviour is identical — because the config was always cloud-shaped.

Challenge 05 — The Rewire

Kill the 2am cron

── BEFORE ────────────────────────────────────── Cron 2am → scan /mnt/reports → INSERT INTO DB Fails silently. Nobody knows until standup. ── AFTER ─────────────────────────────────────── File uploaded → S3 ObjectCreated event → ECS task spawns → handler.js processes CSV → INSERT INTO reporting.daily_summary (idempotent) → compatibility shim (legacy consumers still work) → file moved to /processed Local: MinIO fires webhook → same handler.js AWS: S3 → SQS → ECS task → same handler.js Zero code changes between environments.

The old reconciliation job ran on a cron at 2am and wrote directly to the reporting database schema with a hardcoded IP. If it failed, nobody knew until morning. We replaced it with an event-driven handler that fires the moment a file lands in the bucket — locally in MinIO, in production via S3. We also wrote a compatibility shim so the five downstream teams who depend on the processed output don't break during the 30-day transition window.

Challenges 09 + 10 — The Safety Net

RTO 30 min · RPO 5 min · Rollback at every stage

🌏

Warm Standby DR

Secondary region in ap-southeast-1 Singapore. RDS read replica promotes in 8 minutes. Route53 flips DNS. Smoke tests confirm at T+22. Detailed minute-by-minute runbook — not just a diagram.

RTO 30 min RPO 5 min
↩️

Phase-Level Rollback

On-prem stays warm for 30 days. DNS is the rollback switch. If new data was written to cloud, we export the delta and replay it before cutting back. Each workload rolls back independently.

webapp: 4 min DB: 20 min recon: 3 min

DR and rollback aren't afterthoughts here — they're part of the migration design. We never decommission on-prem infrastructure during the cutover window. The rollback plan includes a printed 4am checklist with exact bash commands for each phase, because when something breaks at midnight, nobody has time to think.
